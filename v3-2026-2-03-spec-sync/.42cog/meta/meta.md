## 项目描述

面向 YouTube 内容创作者的**全流程辅助工具**，核心价值是帮助创作者**找到好选题、了解竞争格局、优化自己的频道**。

不同于简单的数据罗列工具，本项目提供：
1. **有分析框架**（四象限、时长-播放矩阵）而非只展示原始数据
2. **可解释的 AI 洞察**（推理链、置信度、数据源关联）
3. **Actionable**（每个洞察都对应具体行动建议）

## 核心功能模块

| 模块 | 功能 | 解决的问题 |
|------|------|-----------|
| **选题发现** | 输入关键词，分析该领域 1000+ 视频 | "我该做什么内容？" |
| **竞品分析** | 视频榜单、频道榜单、模式洞察 | "这个领域谁做得好？怎么做的？" |
| **频道诊断** | 输入频道链接，获得评分和改进建议 | "我的频道哪里需要改进？" |
| **监控追踪** | 关键词监控、趋势追踪、定时采集 | "市场有什么新变化？" |
| **发布辅助** | 批量上传、定时发布 | "如何高效发布视频？" |

## 目标用户

主要面向**中文 YouTube 创作者**，特别是：
- 做养生、教程、美食、旅行等垂类内容
- 目标受众在新加坡、马来西亚、台湾、香港等华语地区
- 想通过数据驱动来提升内容表现

## 核心价值主张

### 数据分析维度

**视频榜单：**
- 🔥 爆款视频（高播放量）
- ⚡ 近期爆款（新发布但增长快）
- 🌲 长青视频（老视频但日均播放高）
- 📈 潜力视频（被低估的内容）
- ⏱️ 长视频榜

**频道榜单：**
- 📊 总播放榜
- ⚡ 平均播放榜
- 🐴 黑马榜（粉丝少但表现好）
- 🚀 高效榜

**筛选维度：**
播放量、地区、时长、粉丝数、发布时间、互动率、AI视频识别

### 可视化设计

#### 1. 内容四象限 (Content Map)

```
        高互动率
           ↑
    💎 粉丝向   ⭐ 爆款型
    (Niche)      (Star)
           ↑
低播放 ←——————————————→ 高播放
           ↓
    ❄️ 冷门型   🚀 破圈型
    (Dog)       (Viral)
           ↓
        低互动率
```

把**播放量 × 互动率**两个维度交叉，产生四种内容类型：
- **Star**：学习标杆
- **Niche**：深耕细分
- **Viral**：优化互动
- **Dog**：避免重复

#### 2. 可解释 AI 洞察系统

```
┌─────────────────────────────────────────────┐
│ 💡 洞察标题                    [展开/折叠]  │
├─────────────────────────────────────────────┤
│ [置信度条 ████████░░ 78%]                   │
│ 数据源: [视频数据] [频道数据] [趋势数据]     │
├─────────────────────────────────────────────┤
│ ┌─────────────┐  ┌─────────────────────┐   │
│ │  可视化图表  │  │    推理链          │   │
│ │  (SVG/图)   │  │  📊 观察 → 权重30% │   │
│ │             │  │      ↓             │   │
│ │             │  │  📈 分析 → 权重50% │   │
│ │             │  │      ↓             │   │
│ │             │  │  ✅ 结论           │   │
│ └─────────────┘  └─────────────────────┘   │
└─────────────────────────────────────────────┘
```

核心设计：
- **置信度条**：用颜色（高/中/低）表示洞察可靠性
- **数据源标签**：点击可高亮关联数据
- **推理链展示**：把 AI 得出结论的过程拆解成步骤
- **权重条**：显示每个推理步骤的贡献度
- **可交互高亮**：源数据 ↔ 洞察 ↔ 依赖项之间联动高亮

#### 3. 时长分布可视化

```
0-1分钟   ████░░░░░░ 供给 12%
1-3分钟   ██████░░░░ 供给 28%
3-5分钟   ████████░░ 供给 15%   ← 平均播放最高
5-10分钟  ██████████ 供给 8%
```

### 套利分析框架

```
有趣度 = 信息价值程度 / 信息传播程度 = 中介中心性 / 程度中心性
```

高有趣度 = 价值被低估 = 套利机会

| 套利类型 | 发现方法 | 适用博主 |
|----------|----------|----------|
| 话题套利 | 关键词网络中介中心性 | 腰部博主 |
| 跨语言套利 | 源市场价值/目标市场供给 | 头部博主 |
| 时长套利 | 时长分桶有趣度 | 小白博主 |
| 频道套利 | 小频道爆款分析 | 小白博主 |
| 趋势套利 | 话题频率变化率 | 头部博主 |

## 编码规约

### 文件命名约定

为避免全局范围的文件名冲突和混淆，项目采用 **统一的唯一编码体系**：

#### 卡片级（Card）
- **格式**：在 YAML 前置元数据中指定 `id: card-<YYYYMMDD>-<序号3位>`
- **示例**：
  ```yaml
  ---
  id: card-20260203-001
  type: decision
  title: 为什么采用综合模式
  ---
  ```

#### 文档级（Document）
- **格式**：`<YYYYMMDD>_<类型>_<主题>.md`
- **类型**：`insight`（洞察）| `log`（决策日志）| `spec`（规约）| `template`（模板）| `guide`（指南）
- **示例**：
  - `2026-02-03_spec_naming-convention.md`
  - `2026-01-09_log_why-abandon-playwright.md`
  - `2026-02-03_insight_file-conflict-solutions.md`

#### 项目级（Project）
- **格式**：`<project-name>-<module>-<phase>`
- **示例**：`v3-2026-2-03-spec-sync`

### 唯一编码的用途

1. **寻址**：AI 能精准定位文件，避免名字相似导致的混淆
2. **校验**：编码中的日期能帮助 AI 检测异常（如时间顺序错乱）
3. **去重**：同一主题的不同版本通过日期区分，避免误删重要版本
4. **追踪**：编码稳定不变，便于跨文档引用和链接管理

### 冲突预防规则

**禁区**（绝对不允许）：
- ❌ 两个文件有完全相同的名字（包括扩展名）
- ❌ 同一文件夹中有多个 `README.md`（必须标识主题）
- ❌ 不同层级中相同的项目名字不加区分（子文件夹内必须包含父信息）

**必做**（强制执行）：
1. 创建任何文件前，用 Grep 检查是否存在同名文件
2. 文件名必须包含日期或唯一标识符
3. 如果发现潜在冲突，主动列出所有候选名字供人类选择

**异常情况**：
- 如果确实需要多个同名文件，格式：`<名字>_v<版本号>.md` 或 `<名字>_<序号>.md`
